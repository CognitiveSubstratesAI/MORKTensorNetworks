H3 detail (audit §9, confirmed vs MORK reachability.jl / Core info_flow_zipper.jl):
`(edge a b)(edge b c)(edge a c)` previously gave 14 byte-prefix pseudo-nodes + 27
edges; correct is 3 symbol-nodes {a,b,c} + 3 edges (a→b, b→c, a→c). `path_compose` now
computes real 2-hop reachability; arity-4 `(rel src dst weight)` decodes the weight onto
the edge. The dense→sparse-output / SoA-arena rewrite stays as the C4 identity decision —
this fix corrects the *relation encoding*, not density.

H6 detail (audit §9.4, vs `packages/Core/lib/ecan/`): (1) no VLTI (Core's AV is a
STI/LTI/VLTI triple; bridge touches only STI); (2) single threshold-gated STI rent vs
Core's two-tier WA+AF rent on STI+LTI; (3) plain-product Hebbian vs Core's affine
`hebbian-conjunction` (both-zero → 0.5 midpoint); (4) no STI conservation vs Core's
`trade-sti!` source-clamp. The spec ends at §6 (L3: the bridge's §7.3 citations are
dangling), so Core is the only ground truth. Resolution: align to Core or supply §7.
